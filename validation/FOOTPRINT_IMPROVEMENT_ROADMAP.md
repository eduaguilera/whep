# WHEP footprint methodology-gap audit

_Auto-generated by the methodology-gap audit workflow (10 dimensions: WHEP vs afsetools / General / Global / FABIO). Each gap was adversarially re-verified against WHEP's real code before ranking._

## Executive summary

WHEP's Leontief footprint engine is generic and already more rigorous than its peers' manual cascades, but only the LAND extension is wired — so every improvement is porting a peer impact LAYER into the existing extensions vector, not rebuilding the engine. Start with the GHG extension (rank 1): WHEP's IPCC Tier 1/2 livestock emissions pipeline is fully built and just needs a builder that aggregates cohort emissions to (area_code, item_cbs_code), applies GWP100, and feeds compute_footprint() — highest leverage, medium effort, no peer even has this. Then fix co-product allocation (rank 2): replace .drop_derived_land_extensions() with price-weighted allocation using the already-built-but-orphaned build_cbs_prices(), so residues/cotton-seed/palm-oil bear land by economic value instead of implicit 1:1 mass. Water and labour (ranks 3-4) are high-leverage new dimensions whose effort is dominated by sourcing external data (Mekonnen-Hoekstra; Labour repo), with FABIO/Global as direct templates. Do NOT regress WHEP's leads — its gridded CROPGRIDS/ha-yr land builders, endogenous feed-in-Z attribution, and explicit draught IO sector all exceed the peers; promoting cropgrids_fallow to the default land source is a one-line win.

## Ranked improvement roadmap

### #1 — Wire a livestock GHG (CO2e) extension into compute_footprint()

- **Dimension:** GHG / GWP extension  

- **Gap kind:** `missing_allocation` | **Leverage:** high | **Effort:** medium | **Confidence:** 0.92


**Why it matters.** WHEP already has a complete IPCC 2019 Tier 1/2 livestock emissions pipeline (enteric CH4, manure CH4, manure N2O) at cohort level, but those emissions never reach the footprint engine. This is the highest-leverage/lowest-effort improvement: it opens an entirely new, policy-relevant footprint dimension at near-zero engine cost because the upstream calculation already exists and compute_footprint() is extension-agnostic. No peer (afsetools/General/FABIO) has actually wired a GHG footprint, so WHEP would leapfrog all of them.


**Concrete steps in WHEP:**

  1. Add build_livestock_ghg_extension(year) in R/footprint_inputs.R that calls build_primary_production() to get heads by (area_code, item_cbs_code).
  1. Pipe heads through prepare_livestock_emissions() (R/livestock_bridge.R) then calculate_livestock_emissions(tier = 2) with a tier = 1 fallback exposed via a method arg per the multi-method convention in CLAUDE.md.
  1. Aggregate cohort-level enteric_ch4_tier2 + manure_ch4_tier2 + manure_n2o_total back to (area_code, item_cbs_code), applying an IPCC AR6 GWP100 lookup (CH4 = 27.9, N2O = 273) to sum to kg CO2e per sector.
  1. Right-join to the IO model labels tibble to produce a numeric extensions vector aligned to (area_code, item_cbs_code) row order; tag output with extension_scope = 'ghg_livestock_co2e'.
  1. Add a crop/soil GHG counterpart later (currently absent entirely); start with livestock since the pipeline is ready.

**Wiring pointer.** Build build_livestock_ghg_extension(year) in R/footprint_inputs.R; feed its numeric vector to compute_footprint() in the inst/scripts/footprints.R loop alongside the existing land extension (lines 95-122). prepare_livestock_emissions() (R/livestock_bridge.R:36-95) stops at species/heads mapping today; extend it with the GWP aggregation step. The engine (R/footprint.R:96-181) needs no change.


**Port from (peer reference).** No peer wires GHG; port the GWP100 conversion idea from General/R/Codes_coefs_fx.R:149-177 (GWP scalar constants) and Fx.R:347-365 (Gases_GWP CO2e conversion), but the emissions source is WHEP's own R/livestock_emissions.R / livestock_enteric.R / livestock_manure.R.

### #2 — Replace mass-implicit allocation with price-weighted co-product allocation

- **Dimension:** Co-product / residue / by-product allocation  

- **Gap kind:** `unwired_layer` | **Leverage:** high | **Effort:** high | **Confidence:** 0.95


**Why it matters.** WHEP currently performs NO economic allocation among co-products: .drop_derived_land_extensions() silently discards residue, livestock-product, and draught rows from the land_fp input, so every derived product (cotton seed, palm oil, straw) inherits land at an implicit 1:1 mass share through the Leontief inverse rather than by economic value. This directly distorts WHICH items bear land impact and by how much. WHEP already has a fully built but orphaned prices pipeline (build_cbs_prices() with residue_price_factor logic), so the allocation layer can be ported into the extension builder rather than rebuilt.


**Concrete steps in WHEP:**

  1. Remove .drop_derived_land_extensions() from get_land_fp_production() (R/footprint_inputs.R:62).
  1. Add private helper .allocate_land_fp_to_coproducts(land_fp, cbs_prices, residue_price_factor) in R/footprint_inputs.R.
  1. Inside it, for each (year, area_code, item_code_impact) group, call build_cbs_prices() (R/prices.R) for per-item prices; set residue rows to price * residue_price_factor (0.1 default).
  1. Compute alloc_share = (tonnes * price) / sum(tonnes * price) within each group and multiply raw impact_u by alloc_share.
  1. Compute the draught share first from livestock-unit data in get_primary_production() and subtract it before product allocation, mirroring the peer cascade.
  1. Feed the allocated impact_u values as the extensions vector to compute_footprint() (shape unchanged).

**Wiring pointer.** The fix plugs into the extensions vector, not the engine. get_land_fp_production() (R/footprint_inputs.R:62) should call .allocate_land_fp_to_coproducts() in place of .drop_derived_land_extensions(); the allocated named numeric vector is still aligned to (area_code, item_cbs_code), so compute_footprint() (R/footprint.R) is untouched.


**Port from (peer reference).** afsetools/R/impact_functions.R:61-119 Allocate_impacts_to_products() (Value_d = Value*Price; Alloc_raw1 = Value_d/sum(Value_d)) and :290-325 Calc_impact_processed(); mirrored in General/R/Fx.R:440-503 and :712-747. FABIO value allocation: fabio/R/09_mrsut.R:58-85 and 10_mrio.R:41-57.

### #4 — Add a labour-hours footprint extension (Labour repo data)

- **Gap kind:** `unwired_layer` | **Leverage:** high | **Effort:** high | **Confidence:** 0.97

It feeds the same generic Leontief engine without modification, so the analytical coverage gain is large. **Wiring pointer.** get_labour_fp_production() in R/footprint_inputs.R feeds the same footprints.R loop that already calls compute_footprint() for land (lines 95-122). One compute_footprint() call per labour category. Engine (R/footprint.R) requires no modification.

### #7 — Defer a biomass/NPP extension pending double-count audit

- **Dimension:** NPP / biomass coefficients (residue & feed mass balance)  

- **Gap kind:** `unwired_layer` | **Leverage:** low | **Effort:** high | **Confidence:** 0.92


**Why it matters.** WHEP applies no harvest index, residue:product, or root:shoot ratio in its land extension; residue land is instead traced endogenously through the Leontief A matrix via CBS residue sectors (Straw, Other crop residues), and .drop_derived_land_extensions() deliberately keeps 'Crop residues' out of the directly-set extension. Adding a parallel biomass-area extension risks DOUBLE-COUNTING land the Leontief already attributes. The peer NPP layer is itself unfinished (hardcoded Weed_NPP_Scaling = 0.1, 'THE CODE HAS TO BE FIXED' in Global/R/npp.r). Low leverage, high effort — lowest priority.


**Concrete steps in WHEP:**

  1. Resolve the design question: separate biomass extension vs. rescaling the crop-land extension by (1 + residue_ratio) — do NOT add both.
  1. Only if the audit shows a genuine gap, build build_residue_land_extension() in R/crop_land_extension.R returning (year, area_code, item_cbs_code, impact_u) using Biomass_coefs (kg_residue_kg_product_FM) and harvest index.

  1. Do not depend on the peer NPP layer's grassland/weed components, which are explicitly incomplete.

**Wiring pointer.** Conceptual plug-in is the caller of compute_footprint() (inst/scripts/footprints.R): a new build_residue_land_extension() in R/crop_land_extension.R would return a tibble shaped like the land extension. ## WHEP strengths — do NOT regress these (port INTO them)

- Cropping-intensity / land scaling: WHEP leads decisively. It offers three per-crop, gridded land builders keyed to (year, area_code, item_cbs_code) — LUH2 cell-level apportionment, MapSPAM multi-cropping intensity division, and CROPGRIDS 2020 physical/harvested ratios — plus AEZ-gated rotational fallow attribution (FAOSTAT item 6640) and an ha-yr occupation metric (build_hayr_land_extension via MIRCA2000 cycle lengths). Peers (afsetools calculate_land_scaling/scale_land) use only a single national herbaceous/woody scaling factor with no per-crop or gridded resolution and no ha-yr metric. Port new layers INTO this; do not regress to the peer's two-category split.

- Land extension definition / tracing rigor: WHEP's native CROPGRIDS and ha-yr builders already exceed FABIO's published baseline (raw FAOSTAT harvested area with no multi-cropping correction, no physical-vs-harvested distinction). FABIO's own CROPGRIDS migration is unfinished (undefined 'land' object, TODO block). The only remaining gap is that the DEFAULT (WHEP_CROP_LAND_SOURCE = 'land_fp') still reads the naive external pin — promoting cropgrids_fallow to default is a one-line env-var change.

- Feed / livestock-product attribution bridge: WHEP fully endogenizes feed-to-livestock attribution in the MRIO Z matrix (.build_use_husbandry populates Z[feed_item, livestock_sector]), so the Leontief inverse propagates crop land -> feed -> livestock product automatically, including re-exports and processing. This is structurally superior to the peers' manual 3-iteration cascade (Global/R/animal_feed.r). The only improvement is upgrading the FCR coefficient table (port Global/R/feed_ratios.r L2 output into .build_bouwman_fcr) — no engine change.

- Energy / traction: WHEP models animal draught as an explicit TJ IO sector distributed to crops by area share, more rigorous than the peers' scalar draught_share allocation shortcut.

- Generic Leontief engine: compute_footprint() (R/footprint.R) is a fully extension-agnostic MRIO solve that traces any numeric extensions vector through endogenous trade/processing/feed. This is more rigorous than General's manual cascade, which is why every improvement above is a PORT of a peer impact LAYER into this engine, never a replacement of it.
